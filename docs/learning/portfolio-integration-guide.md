## 0. 先建立一个大画面

我们要做的事，一句话：**把你电脑里分散的几个网页项目，合并成一个网站，放到一台你买的服务器上，让全世界用你的域名都能打开。**

你可以把它想象成开一家"商场"：

- **服务器** = 你租的一栋楼（有水电、有地址）
- **域名** = 这栋楼挂的招牌名字（别人记招牌，不用记门牌号）
- **Nginx** = 商场大门的前台/导购（所有人从这一个门进，前台指引你去哪个店）
- **各个项目（RAG、Function Calling、Nexus、学习站、刷题）** = 商场里的各个店铺
- **Docker** = 每个店铺都是一个"标准集装箱店面"，搬到任何商场都能原样开张

下面逐个拆开讲。

## 1. 网页是怎么"被别人打开"的（最基础）

当别人在浏览器输入你的网址、按回车，背后发生了这些事：

```
1. 浏览器问 DNS：「huzhiming.com 这个名字，对应哪台机器？」
2. DNS 回答：「对应 IP 地址 1.2.3.4」（你服务器的公网地址）
3. 浏览器连上 1.2.3.4，说：「把首页发我」
4. 服务器上的程序把网页内容发回来
5. 浏览器把收到的 HTML/CSS/JS 画成你看到的页面
```

几个关键名词：

- **IP 地址**：每台联网机器的"门牌号"，像 `1.2.3.4`。你买服务器送的那个 `1 IPv4 公网地址`就是它。
- **域名（Domain）**：给 IP 起的好记名字，比如 `huzhiming.com`。人记名字，机器用 IP。
- **DNS（域名系统）**：一个"通讯录"，负责把域名翻译成 IP。你在域名控制台加的 **A 记录**，就是往这个通讯录里写一条「我的域名 → 我的服务器 IP」。
- **端口（Port）**：一台机器上可以同时跑很多服务，端口就是"几号窗口"。网站默认用 **80（HTTP）** 和 **443（HTTPS）**。

> 类比：域名是公司名，IP 是公司地址，端口是前台几号窗口，DNS 是查号台。

## 2. 静态网站 vs 动态后端（为什么 AI demo 不能"纯静态"）

这是你最早问"只要域名不要服务器能不能做网站"时，最该理解的一点。

### 静态网站

内容是**写死的文件**（HTML/CSS/JS/图片）。服务器只负责把文件原样发给浏览器，不"思考"。
- 例子：博客文章、公司介绍页、你的门户首页、学习站点。
- 好处：可以放在免费的"静态托管"上（GitHub Pages 等），**只要域名不要自己的服务器**。

### 动态后端

每次请求都要服务器**现场运算**再返回结果。
- 例子：你的 RAG 问答——用户问一句，服务器要去调大模型、查向量库、组织答案，**每次答案都不一样**。
- 这种**必须有一台一直在跑的服务器**，纯静态托管做不了。

### 为什么 AI demo 不能把密钥放前端

有人会想：那我让浏览器里的 JS 直接调大模型 API 不就行了？**不行**：
- 调 API 需要你的 **API Key**（相当于你账户的密码 + 付费凭证）。
- 前端 JS 是公开的，任何人按 `F12` 就能看到里面的 Key。
- 别人拿到 Key 就能盗刷你的额度、花你的钱。

**所以正确做法**：Key 只放在**服务器后端**，浏览器把问题发给你的后端，后端拿着 Key 去调 API，再把结果转回浏览器。浏览器**永远看不到 Key**。这也是我们设计里"所有后端只监听 127.0.0.1、由 Nginx 转发"的根本原因。

## 3. Nginx 与"反向代理"（为什么需要一个总入口）

你的网站里其实有好几个独立程序在跑：RAG 后端、FC 后端、Nexus 后端、刷题后端、还有一堆静态文件。它们各自监听不同端口（8001、8002…）。

但访客只会用**一个域名**进来。谁来把"进来的人"分发到正确的程序？——**Nginx**。

### Nginx 是什么

一个高性能的"网站服务器 + 流量调度员"。在我们的设计里它干三件事：

1. **当总入口**：独占 80/443，是唯一对公网开放的程序。
2. **发静态文件**：首页、学习站这些静态内容，Nginx 直接发。
3. **反向代理**：动态请求按路径转发给后面的后端程序。

### 什么是"反向代理"

"代理"就是中间人。**反向代理** = 站在服务器这一侧的中间人：

```
访客 → Nginx → /rag        转发给→ 127.0.0.1:8001 (RAG 后端)
            → /fc         转发给→ 127.0.0.1:8002 (FC 后端)
            → /nexus      转发给→ 127.0.0.1:8003 (Nexus 后端)
            → /doctomd    转发给→ 127.0.0.1:8004 (DocHub 后端)
            → /iconforge  转发给→ 127.0.0.1:8005 (IconForge 后端)
            → /learn/     直接发 frontends/nexus-learning-web/dist
            → /           直接发 frontends/portfolio/dist（React 门户）
```

好处：
- 对外只暴露一个端口（安全，后端都藏在 127.0.0.1 内部）。
- 访客感觉是"一个站"，其实背后是好几个程序。
- 统一在这里处理 HTTPS 证书。

> **127.0.0.1** 是"本机自己"的特殊地址（也叫 localhost）。后端只监听它，意味着**只有同一台机器上的 Nginx 能访问它们，外网直接连不进来**——这是一道安全墙。

## 4. Docker 与 docker-compose（为什么本地=线上）

### 痛点

"在我电脑上能跑，搬到服务器就报错"——因为两台机器的 Python 版本、依赖库、系统环境都不一样。

### Docker 是什么

把一个程序**连同它需要的整个环境**（操作系统层、Python、依赖库、配置）打包成一个**镜像（image）**，运行起来就是一个**容器（container）**。这个容器搬到任何装了 Docker 的机器上，**行为完全一致**。

> 类比：集装箱。不管货物是什么，集装箱尺寸标准，任何货轮、卡车都能直接装运，不用关心里面是啥。

- **镜像（image）**：打包好的"店面模板"（只读）。
- **容器（container）**：用模板开起来的"营业中的店"（运行实例）。
- **volume（数据卷）**：容器是可以随时拆了重建的，但有些数据（SQLite 数据库、Chroma 向量库）不能丢，就存到 volume 里，**容器重建数据还在**。

### docker-compose 是什么

我们有 5 个服务（nginx + 4 个后端）。一个个手动启动太麻烦。**docker-compose** 用一个 `docker-compose.yml` 文件描述"我有哪些服务、各自怎么跑、谁连谁"，然后：

```
docker compose up -d        # 一条命令把所有服务启动
docker compose down         # 一条命令全部停掉
```

最大的好处：**本地开发和线上服务器用同一份 compose 文件**。本地 `up` 跑通了，搬到服务器还是 `up`，环境一模一样。"部署"几乎就等于"在服务器上再 up 一次"。

- `restart: always`：配置项，容器崩了自动重启，开机自启动。

## 5. iframe 嵌入（方案 C 的核心折中）

你的 RAG/FC/Nexus 这几个 demo，界面风格各不相同（它们是不同时期写的）。要让它们"看起来在同一个站里"，有两条路：

- **重写**：全部改成统一的 React 界面（漂亮但工作量巨大）= 方案 A
- **嵌入**：保留原界面，用 iframe 把它"镶"进统一外壳里 = 方案 C（我们选的）

### iframe 是什么

HTML 里的一个标签 `<iframe>`，作用是**在一个网页里嵌入另一个网页**，像开一个"画中画窗口"。

```
门户外壳页面 (/rag)
┌─────────────────────────────────┐
│  [统一顶部导航：首页 AI作品 学习 个人] │  ← 外壳提供
├──────────────────┬──────────────┤
│                  │  技术说明面板  │
│   iframe          │  · 解决什么问题│  ← 外壳提供
│   （嵌入原 RAG     │  · 用了什么技术│
│    demo 界面）     │  · GitHub 链接 │
│                  │              │
└──────────────────┴──────────────┘
```

好处：几乎不改原 demo，快速上线。
代价：iframe 有些细节要调（高度自适应、内外样式隔离），所以它是"先上线"的折中，以后可以逐个换成原生 React。

### iframe 内外主题同步

因为各 demo 原本风格不统一，2026-06 又给门户加了多套主题（mono-light / light / deepblue / cyber / machine）。如果 iframe 内部不变色，访客会感觉很割裂。

解决方案：**同源 iframe + localStorage + storage 事件**：

1. 门户切主题时写入 `localStorage.setItem('ai-demos-theme', theme)`；
2. 同源 iframe 自动收到 `storage` 事件；
3. iframe 内脚本设置 `data-demo-theme` 属性；
4. 各 demo 后端使用统一的 `--d-*` token 调色板，自动变色。

这样无需 postMessage、无需刷新，各 demo 内部就能实时跟随门户主题。

## 6. SSE：让"多智能体协作"看得见（Nexus 的亮点）

Nexus 是你最有技术含量的作品。它的价值在于**多个智能体（Agent）协作**完成一个任务。但如果只显示一个最终答案，访客根本看不出"多智能体"在哪。

我们要让前端**实时显示中间过程**：

```
你问：帮我查 RAG 评估方法并写个示例

Planner   正在拆解任务……          ✓
Retriever 正在检索知识库……        ✓
Executor  正在执行代码……          ✓
Summarizer 正在汇总……            ✓
Critic    正在评分……  得分 0.85   ✓

最终答案：……
```

### SSE 是什么

**SSE（Server-Sent Events，服务器发送事件）**：一种让**服务器主动、持续地往浏览器推送消息**的技术。

- 普通请求是"一问一答"：浏览器问一次，服务器答一次，结束。
- SSE 是"问一次，服务器陆续推很多条"：浏览器发起一次连接，服务器每完成一个 Agent 步骤就推一条消息过去，前端收到就更新画面。

> 类比：普通请求像寄信（来回一次）；SSE 像直播弹幕（建立一次连接后，主播持续往你屏幕推内容）。

这就是为什么用户能"看见" Planner → Retriever → Executor 一步步执行，而不是干等一个最终结果。

## 7. HTTPS 与 TLS 证书（地址栏那把小锁）

- **HTTP**：明文传输，别人能在中途偷看你和网站之间的数据。
- **HTTPS**：加密传输，地址栏有把小锁。现在不做 HTTPS 的网站，浏览器会标"不安全"，很掉价。
- **TLS 证书**：实现 HTTPS 需要一张"证书"，用来加密 + 证明"这个域名真的是你的"。
- **Let's Encrypt**：一个**免费**自动发证书的机构，用工具（如 certbot）几条命令就能给你的域名签发证书，到期自动续。

我们的 Nginx 负责"TLS 终止"——意思是加密解密都在 Nginx 这层做完，后面的后端程序之间用普通 HTTP 通信即可（反正都在同一台机器内部）。

## 8. 把整条链路串起来

一次完整的访问（以"访客用 RAG 问答"为例）：

```
1. 访客浏览器输入 https://你的域名/rag
2. DNS 把域名解析成你首尔服务器的 IP
3. 请求到达服务器 443 端口 → Nginx 接住
4. Nginx 用 TLS 证书解密，看路径是 /rag
5. Nginx 把请求转发给容器内 127.0.0.1:8001（RAG 后端）
6. RAG 后端拿着藏在服务器上的 API Key，调通义千问 + 查 Chroma
7. 后端把答案返回 Nginx → Nginx 加密 → 发回浏览器
8. 访客看到答案。整个过程 API Key 从没离开服务器
```

部署时你要做的事，按顺序：

```
服务器侧：重装 Ubuntu → 加 swap → 装 Docker → 放行 80/443 端口
域名侧：  加 A 记录指向服务器 IP
代码侧：  git clone 仓库 → 放好 .env → docker compose up -d --build
证书侧：  certbot 申请 HTTPS 证书
验证：    浏览器逐个路径点开测试
```

## 9. 名词速查表

| 名词 | 一句话解释 |
|---|---|
| IP 地址 | 机器的门牌号，如 1.2.3.4 |
| 域名 | IP 的好记别名，如 huzhiming.com |
| DNS / A 记录 | 把域名翻译成 IP 的通讯录 / 通讯录里的一条记录 |
| 端口 | 一台机器上的"几号服务窗口"，网站默认 80/443 |
| 静态网站 | 内容写死的文件，可放免费托管，无需自己服务器 |
| 动态后端 | 每次请求现场运算，必须有一直在跑的服务器 |
| Nginx | 网站总入口 + 静态发文件 + 反向代理 |
| 反向代理 | 站在服务器侧的中间人，按路径把请求分发给不同后端 |
| 127.0.0.1 / localhost | "本机自己"，后端只监听它=外网连不进来 |
| Docker 镜像 / 容器 | 打包好的环境模板 / 跑起来的运行实例 |
| docker-compose | 用一个文件编排多个容器，一条命令全启动 |
| volume | 容器外的持久化数据存储，容器重建数据不丢 |
| iframe | 在一个网页里嵌入另一个网页（画中画） |
| SSE | 服务器持续主动推消息给浏览器（像直播弹幕） |
| HTTPS / TLS 证书 | 加密传输 / 实现加密所需的证书 |
| Let's Encrypt | 免费自动发 TLS 证书的机构 |
| swap | 用硬盘临时充当内存，防止小内存机器被打满崩溃 |

## 10. 配套学习路径建议

跟着做这个项目，你会顺带学到一串 AI 工程师该会的技能，建议这样补：

1. **HTTP 基础**：请求/响应、状态码、GET/POST —— 配合实操理解。
2. **Nginx 基础**：反向代理配置、location 块、静态托管。
3. **Docker 基础**：镜像/容器/volume，写一个最小 Dockerfile，再读懂 docker-compose.yml。
4. **Linux 基础**：SSH 连服务器、文件操作、查看进程/内存、看日志。
5. **HTTPS / certbot**：申请证书、自动续期。
6. **SSE / 流式接口**：理解和普通接口的区别，能讲清楚为什么用它做 Agent 可视化。

这些每一项都是面试常问点，而你是"在真实项目里用过"，比背概念强得多。

## 11. Phase 1 实操回顾(把踩的坑变成学习案例)

Phase 1 已经完成,整套门户在本地 `docker compose` 跑起来了。下面把实现过程中真实遇到的问题拎出来——这些就是面试时能讲的"工程细节"。

### 11.1 现在你能动手看的东西

```bash
# 1. 启动 Docker Desktop(等引擎就绪)
# 2. 构建前端
bash deploy/build-frontends.sh
# 3. 起全部容器
docker compose -f deploy/docker-compose.yml up -d --build
# 4. 浏览器 http://127.0.0.1:8080
```
看完 `docker compose ps`(看三个容器)、`docker logs deploy-rag-1`(看后端日志)、`docker logs deploy-nginx-1`,再对照 `deploy/nginx/nginx.conf` 理解请求怎么被分发。

### 11.2 坑 1:node_modules 被提交进 git(68MB)

新建的 `frontends/portfolio` 没有 `.gitignore`,`npm install` 后 `git add` 把 3700+ 个依赖文件全提交了。
- **教训**:任何前端工程,建好第一件事就是加 `.gitignore`(node_modules/、dist/)。
- **怎么修的**:因为分支没推送过,用 `git reset --mixed` 把那两个臃肿提交退回,补 `.gitignore` 后重新只提交源码。学到:`git reset --soft/--mixed/--hard` 的区别。

### 11.3 坑 2:子路径反向代理(最值得讲的一个)

RAG/FC 的网页里 JS 是往**绝对路径** `/chat` 发请求的。直接访问后端(8001)没问题,但一旦经 Nginx 挂到 `/rag/` 子路径下(在 iframe 里),浏览器在 `/rag/` 页面发 `/chat` 会打到**站点根**而不是 `/rag/chat`,功能就废了。
- **修法**:把 fetch 路径从 `/chat` 改成相对的 `chat`。浏览器在 `/rag/` 下解析相对路径 `chat` → `/rag/chat` → Nginx 的 `location /rag/ { proxy_pass http://rag:8001/; }` 去掉前缀 → 后端收到 `/chat`。
- **关键知识点**:`proxy_pass` 结尾带不带 `/` 决定要不要剥前缀;相对 URL 相对"当前文档路径"解析(所以 iframe 的 src 必须带结尾斜杠 `/rag/`)。

### 11.4 坑 3:Windows 换行符(CRLF)

Git 在 Windows 默认把 `.sh`、Dockerfile 检出成 CRLF。bash 执行带 `\r` 的脚本会报 `'\r': command not found`,Linux 容器里也可能出错。
- **修法**:加 `.gitattributes`,对 `*.sh`/`Dockerfile`/`*.conf` 强制 `eol=lf`。这是跨平台协作的标准做法。

### 11.5 坑 4:Docker 环境

- Docker CLI 装了不代表引擎在跑——**Docker Desktop 必须先启动**(否则报找不到 `dockerDesktopLinuxEngine` 管道)。
- 国内拉 Docker Hub 镜像易超时:先 `docker pull nginx:1.27-alpine`、`docker pull python:3.12-slim` 预缓存,或在 Docker Desktop 配镜像加速器。
- 2G 内存小服务器:别在本机跑大模型,走云端 API;加 swap。

### 11.6 这次用到的工程流程(也是能讲的)

需求脑暴(brainstorming)→ 写设计文档(spec)→ 写实现计划(plan,拆成可验收任务)→ 子代理逐任务实现 + 评审 → 整分支总评审 → 收尾。每个任务 TDD/验证 + 频繁提交。这套"先设计后实现、带评审关卡"的习惯,本身就是工程素养。
